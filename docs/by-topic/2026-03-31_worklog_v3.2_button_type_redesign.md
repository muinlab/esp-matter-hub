## 1. 개요

이 세션에서는 **슬롯 모델 전면 재설계**를 통해 v3.1의 문제점을 해결했습니다. 주요 변경사항:

- **button_type 열거형** 도입 (ONOFF=0, LEVEL=1, ONLYON=2)
- **SendSignalWithRaw 파라미터 단순화** (4개로 축소)
- **signal_id_a / signal_id_b** 구조로 슬롯-신호 바인딩 명확화
- **웹 API 보안** (API 키 인증 + NVS 저장)
- **Signal Buffer 리네이밍** (cache → buffer, ref_count + timestamp 추가)
- **새 Matter 커맨드** (SyncBuffer 0x0C, FactoryReset 0x0D)

## 4. 웹 API 보안 (v3.2 신기능)

### 4.1 API 키 인증

**메커니즘**:
```c
1. 부팅 시 API 키 생성 (또는 NVS에서 로드)
2. 랜덤 16진 문자열 (예: "a3f2b1c8d9e4f7a2")
3. 직렬 로그에 표시
4. HTTP 헤더: X-Api-Key: {key}
```

**코드**:
```c
static void generate_api_key() {
    nvs_handle_t handle;
    esp_err_t err = nvs_open(kNvsNamespaceWeb, NVS_READONLY, &handle);
    if (err == ESP_OK) {
        size_t len = sizeof(s_api_key);
        err = nvs_get_str(handle, kNvsKeyApiKey, s_api_key, &len);
        nvs_close(handle);
        if (err == ESP_OK && s_api_key[0] != '\0') {
            return;  // 기존 키 로드
        }
    }

    // 새 키 생성
    uint32_t r0 = esp_random();
    uint32_t r1 = esp_random();
    snprintf(s_api_key, sizeof(s_api_key), "%08lx%08lx", r0, r1);
    save_api_key_to_nvs();  // NVS 저장
}
```

### 4.2 NVS 저장 스키마

| Namespace | 키 | 타입 | 설명 |
|-----------|-----|------|------|
| `web_config` | `api_key` | String | 16자 API 키 (부팅 시 생성 또는 로드) |

**부팅 시 동작**:
```
1. NVS web_config:api_key 읽기
2. 존재 → 로드 후 사용
3. 미존재 → 새 키 생성 → 저장
4. 직렬 로그 출력:

```

## 5. button_type 재설계 (Deep Interview → Ralplan → Autopilot)

### 5.1 개념 정의

**button_type 열거형** (v3.2 신규):
```c
typedef enum button_type {
    BUTTON_TYPE_ONOFF  = 0,  // On/Off 토글 (2가지 신호)
    BUTTON_TYPE_LEVEL  = 1,  // 레벨 조정 (Up/Down, 2가지 신호)
    BUTTON_TYPE_ONLYON = 2,  // On만 지원 (1가지 신호)
} button_type_t;
```

**슬롯 구조 변경**:
```c
typedef struct bridge_slot_state {
    uint8_t slot_id;
    uint16_t endpoint_id;
    button_type_t button_type;        // NEW: 버튼 타입
    char display_name[40];
    uint32_t signal_id_a;             // ON / UP / 신호
    uint32_t signal_id_b;             // OFF / DOWN / 미사용(0)
} bridge_slot_state_t;
```

### 5.2 디바이스 레지스트리 제거

**v3.1**:
```
앱 → SendSignalWithRaw(device_id, ...)
→ 허브가 device_id 레지스트리 관리
→ device_type 고정 (문제: 복잡성)
```

**v3.2**:
```
앱 → SendSignalWithRaw(signal_id, ...)
→ 허브가 signal 버퍼에 저장
→ 슬롯 = button_type(명시적) + signal_id_a/b
```

**이점**:
- 레지스트리 제거 → 코드 단순화
- 슬롯 설정 시 button_type 명시 → 명확성 증대
- signal_id만 추적 → NVS 저장 용량 절감

### 5.3 SendSignalWithRaw 단순화 (4 params)

**v3.1** (복잡):
```
SendSignalWithRaw(
    device_id,        // 디바이스 등록/추적 용
    is_level,         // 레벨 타입 여부
    high_low,         // up/down 구분
    signal_id,        // 신호 ID
    carrier_hz,
    repeat,
    ticks,
    tick_count
)
```

**v3.2** (단순):
```
SendSignalWithRaw(
    signal_id,        // 신호 ID (NVS 캐시 용도)
    carrier_hz,       // 반송파 주파수
    repeat,           // 반복 횟수
    ticks,            // timing payload (uint16_t[])
    tick_count
)
```

**호출 예시**:
```

  --signal-id 12345 \
  --carrier-hz 38000 \
  --repeat 1 \
  --ticks "[9265,4490,636,539,...]"
```

## 6. Signal Buffer (cache → buffer 리네이밍)

### 6.1 용어 정리

**v3.1**: "Signal Cache" (혼동 유발: 앱의 Signal vs Hub의 Cache)
**v3.2**: "Signal Buffer" (명확: Hub의 16-entry LRU 버퍼)

### 6.2 구조 확장

**v3.1**:
```c
typedef struct signal_buffer_entry {
    uint32_t signal_id;
    uint32_t carrier_hz;
    uint8_t repeat;
    rmt_item32_t items[64];
    size_t item_count;
    uint32_t last_used;
    bool valid;
} signal_buffer_entry_t;
```

**v3.2** (추가 필드):
```c
typedef struct signal_buffer_entry {
    uint32_t signal_id;
    uint32_t carrier_hz;
    uint8_t repeat;
    rmt_item32_t items[64];
    size_t item_count;
    uint32_t last_used;
    bool valid;

### 6.3 NVS 영속성

**정책**: 버퍼가 가득 찼을 때만 NVS에 저장 (LRU 제거 항목)

        if (!s_signal_buffer[i].valid) continue;

        snprintf(key, sizeof(key), "c%u", s_signal_buffer[i].signal_id);

        // (구현 상세는 ir_engine.cpp 참조)

### 7.2 구현 (app_main.cpp)

#include <esp_sntp.h>

void sntp_sync_time() {
    esp_sntp_setoperatingmode(SNTP_OPMODE_POLL);
    esp_sntp_setservername(0, "pool.ntp.org");
    esp_sntp_init();

    // NTP 동기화 대기
    time_t now = time(NULL);
    int retry_count = 0;
    while (now < 24*3600*1000 && retry_count++ < 20) {
        vTaskDelay(pdMS_TO_TICKS(100));
        now = time(NULL);
    }
}
```

## 9. 새 Matter 커맨드 (v3.2)

### 9.1 SyncBuffer (0x0C)

**목적**: Signal Buffer의 모든 유효한 항목을 강제로 NVS에 저장

**파라미터**: 없음

**응답**: DefaultSuccessResponse (0x00)

**사용 사례**:
```
앱 → 여러 신호 학습 및 SendSignalWithRaw
→ 허브 리부팅 예상
→ SyncBuffer 호출
→ 모든 버퍼 항목 → NVS 저장
```

### 9.2 FactoryReset (0x0D)

**목적**: 허브 초기화 (모든 NVS 데이터 삭제)

**파라미터**: 없음

**응답**: DefaultSuccessResponse (0x00)

**동작**:
```c
esp_err_t factory_reset() {
    nvs_flash_erase();              // 모든 NVS 파티션 삭제
    esp_restart();                  // 재부팅
}
```

**주의**: 슬롯 설정 + 캐시 + API 키 모두 삭제

## 11. pair_auto SSID 감지 수정

### 11.1 문제

**원본**: 부팅 시 `pair_auto` 스크립트가 WiFi SSID를 자동 감지하려고 시도

**macOS 프라이버시**: 권한 없이 WiFi 목록 접근 불가

### 11.2 해결책

**수정 방향**:
1. WiFi SSID 자동 감지 제거
2. BLE PASE 단계에서 사용자가 명시적으로 WiFi 크레덴셜 입력
3. 또는 사전에 설정된 환경 변수로 주입

**수정 후 흐름**:
```
부팅 → BLE 대기
→ 사용자 앱 연결
→ QR/Manual Code 스캔
→ WiFi SSID/PW 입력
→ BLE PASE (PIN)
→ WiFi 크레덴셜 수신
→ 허브 WiFi 연결
```

## 12. SyncBuffer 개선 — NVS 저장 후 버퍼 비우기

### 12.1 개선 사항

**v3.1**: SyncBuffer 실행 후 버퍼 유지
**v3.2**: SyncBuffer 실행 후 버퍼 전체 invalid 처리

### 12.2 구현

```c
void ir_engine_sync_buffer_to_nvs() {
    // 1. 버퍼 → NVS 저장
    persist_buffer_to_nvs();

    // 2. 버퍼 전체 비우기
    for (int i = 0; i < SIGNAL_BUFFER_SIZE; ++i) {
        s_signal_buffer[i].valid = false;
        s_signal_buffer[i].signal_id = 0;
        s_signal_buffer[i].ref_count = 0;
    }

    ESP_LOGI(TAG, "Buffer synced and cleared");
}
```

### 12.3 목적

- **메모리 효율**: 버퍼 재사용 준비
- **상태 명확성**: 저장된 신호는 NVS에만 존재
- **충돌 방지**: 버퍼 + NVS 중복 저장 제거

## 13. DumpNVS (0x0E) 신규 커맨드

### 13.1 목적

허브의 모든 NVS 저장 신호를 앱에 제공

### 13.2 동작 흐름

```
1. SyncBuffer 실행 (버퍼 → NVS 저장 + 버퍼 비움)
2. ir_cache 네임스페이스의 모든 신호 읽기
3. JSON 형식으로 BufferSnapshot(0x0007) 속성에 기록
```

### 13.3 ir_engine 새 함수

```c
int ir_engine_read_all_nvs_signals(json_t *output_array) {
    nvs_iterator_t it = nvs_entry_find(kNvsNamespaceIrCache, NVS_TYPE_BLOB);
    int count = 0;

    while (it != NULL) {
        nvs_entry_info_t info;
        nvs_entry_info(it, &info);

        // signal_id, carrier_hz, repeat, item_count, ref_count, last_seen_at 읽기
        // JSON 객체 생성 후 output_array에 추가
        count++;

        it = nvs_entry_next(it);
    }

    nvs_release_iterator(it);
    return count;
}
```

### 13.4 BufferSnapshot 속성 확장

**v3.1**: 1024 바이트
**v3.2**: 2048 바이트 (더 많은 신호 저장)

### 13.5 커맨드 사용 예시

```bash
chip-tool iermanagement dump-nvs 1 10
# 후속: BufferSnapshot(0x0007) 읽기
chip-tool any read-by-id 0x1337FC01 0x0007 1 10
```

## 14. LearnedPayload에 학습된 IR 데이터 포함

### 14.1 개선 사항

**v3.1**: LearnState + 상태만 제공
**v3.2**: READY 상태일 때 **carrier + ticks(hex)** 포함

### 14.2 LearnedPayload JSON 형식

```json
{
  "state": 2,              // 0=IDLE, 1=IN_PROGRESS, 2=READY, 3=FAILED
  "elapsed": 3200,         // 학습 시간 (ms)
  "timeout": 15000,        // 타임아웃 (ms)
  "last_id": 0,            // 마지막 신호 ID
  "rx": 1,                 // 신호 수신 여부
  "len": 48,               // RMT 아이템 개수
  "quality": 48,           // 신호 품질 (0-100)
  "carrier": 38000,        // 캐리어 주파수 (Hz)
  "ticks": "A801F40238025803..."  // uint16 LE hex (SendSignalWithRaw와 동일 형식)
}
```

### 14.3 ticks 형식 (uint16 LE)

- 2바이트 단위로 인코딩 (마이크로초)
- 예: `A801F402` = `[0x01A8, 0x02F4]` = `[424, 756]` (마이크로초)
- SendSignalWithRaw의 BYTES 파라미터와 정확히 동일

### 14.4 ir_engine 새 함수

```c
// LearnedPayload JSON 생성 시 ticks 포함
void ir_engine_get_learned_ticks(char *ticks_hex_buf, size_t bufsize) {
    // s_learn_payload의 RMT 아이템을 uint16 LE 바이트로 변환
    // hex string으로 버퍼에 기록
}
```

### 14.5 LearnedPayload 버퍼 확장

**v3.1**: 512 바이트
**v3.2**: 1280 바이트 (더 긴 신호 데이터 수용)

### 14.6 앱 개발 흐름

```
1. StartLearning(0x00) 호출
2. LearnState == READY 폴링
3. LearnedPayload 읽기 → carrier + ticks 획득
4. 앱에 저장 또는 즉시 SendSignalWithRaw에 사용
   - 학습된 ticks를 그대로 복사해서 사용 가능
   - NVS 저장 전 앱이 미리 테스트 가능
```
